B · 能力死链

Qwen 多参考图（★一致性核心）在飞书入口是死的

双重原因叠加：① aeon-spark 镜像未预置 Qwen-Image-Edit 模型，走 qwen 必报缺模型；② 更要命——spark_gen.py:140 写死 mode="flux"，飞书里就算带参考图也永远只走文生图，qwen 路径根本调不到。

正解：下 Qwen 三件套放入 aeon models/ + 让 spark_gen 检测到参考图时传 mode="auto"（当前飞书还不收图片，需先补图片消息支持）。
证据 plugins/spark_gen.py:140 · comfyui_client.py:41 注释 · progress/06 待办④
C · 文档误导 · 幸而无害

FLUX_VARIANT=klein9b 是无效值

progress/06 与会话记录写 FLUX_VARIANT=klein9b，但代码只认 dev（_flux_key = "flux_dev" if …=="dev" else "flux"）。写 klein9b 落到默认 flux——恰好就是 Klein-9B，结果对、理由错。.env.spark.example 已修正为「置 dev / 默认空」，但两份 progress 仍留误导写法。

正解：把两处 progress 文档里的 =klein9b 改成"留空即 Klein-9B"，避免后人以为它是开关。
证据 comfyui_client.py:59 · progress/06:58
D · 维护风险 · 长期漂移

client 是"复制"不是"共享"

现在两份 MD5 相同，但它们是两个仓库里的两个物理文件，无 symlink / 子模块 / import 共享。AIGC-spark 哪天改工作流（换 Klein 新版、加节点），mgj-spark 不会自动跟。当下完美契合，长期会静默漂移。

正解：约定 AIGC-spark 为权威源 + 一条 sync 校验（比对 MD5，不一致告警），或抽成共享包 import。
证据 md5sum 双文件一致 · 物理路径分属两仓库

代码层，它逐字节契合；运行层，它把命押在一个手动起、重启就没的容器上。
补上那一次 SKIP_MODEL_DOWNLOAD 重启，"契合"才从纸面落到地面。
